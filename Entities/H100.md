# H100

NVIDIA Hopper datacenter GPU (compute capability 9.0); the production workhorse before Blackwell.

## Synthesis


NVIDIA's H100 is the Hopper-architecture datacenter GPU that serves as both the production training substrate for large-scale MoE models and the primary target for instruction-level micro-architectural research. At the systems level, MoE Parallel Folding (shipped in Megatron-Core) achieves 49.3% MFU on Mixtral 8x22B and 39.0% MFU on Qwen2-57B-A14B by decoupling the parallelism mappings of attention and MoE layers, scaling to 1,024 H100s at 128K sequence length. At the kernel level, the HKUST/HIT microbenchmark study provides the missing micro-architectural data sheet for three Hopper-novel features — Tensor Memory Accelerator (TMA), Distributed Shared Memory (DSM), and DPX dynamic-programming instructions — alongside FP8 Tensor Core characterization across three levels (instruction, Transformer Engine library, LLaMA application). A practical finding from that study: FP8 speedups are substantially below peak-rate projections because Softmax, GeLU, and DotProductAttention are not FP8-quantized in the current Transformer Engine, meaning kernel authors must account for data-format-conversion overhead when targeting H100.


## Observations

- MoE Parallel Folding's headline MFU numbers are measured on H100: **49.3% MFU on Mixtral 8×22B and 39.0% MFU on Qwen2-57B-A14B** with 5-D hybrid parallelism (TP × EP × CP × DP × PP). Framework scales to 1,024 H100 GPUs at 128K sequence length. — [[2025-10-28-moeparallel-folding-heterogeneous-parall]]
- NVIDIA H100 shipped with an `mma.sp` (sparse instruction) bug causing intermittent SDC in production; fixed via driver update 535.288.01 (Linux). NVIDIA Xid 94 = contained ECC error (affects only the triggering process); Xid 95 = uncontained ECC error (affects all GPU processes). — [[2026-01-26-静默数据损坏-sdc-ai-infra-的隐性杀手]]
- Primary microbenchmark target in HKUST's Hopper-architecture study — instruction-level latency/throughput data across Tensor Core precisions (FP8/BF16/TF32/FP64), TMA async copy, DSM SM-to-SM, DPX dynamic-programming primitives, plus application-level LLaMA-7B/13B/70B generation comparison. — [[2026-01-15-benchmarking-and-dissecting-the-nvidia-h]]
- JAX Scaling Book (Ch. 12, Dec 2025) provides a comprehensive H100 hardware spec reference: 132 SMs at 1.59/1.98GHz (base/boost), each with 4 subpartitions; 256kB SMEM + 256kB registers per SM; 50MB L2 (~5.5 TB/s); 80GB HBM at 3.35 TB/s; peak 990 bf16 TFLOPs/s (Tensor Core) vs. 66 TFLOPs/s (CUDA cores). Critical intensity for H100 fp16 ≈ 295 — need ~280 token batch to be compute-bound. Node-level NVLink 4.0: 18 links × 25 GB/s = 450 GB/s per GPU, 4 NVSwitches per 8-GPU node. — [[2025-12-11-how-to-think-about-gpus-how-to-scale-you]]
- SXM5 variant has 132 SMs (four 4-quadrant SMs per GPC; 8 GPCs; some SMs fused off from 144 theoretical), each SM containing 4 warp schedulers (128 threads true parallel per cycle, 2048 threads concurrent per SM); TMA introduced for async global↔shared memory transfers and swizzling for bank-conflict reduction. — [[2025-10-12-inside-nvidia-gpus-anatomy-of-high-perfo]]
- inference.ai GPU calculator includes H100 PCIe and H100 SXM at 80 GB VRAM; for a sample 3.77B-param BF16 model, H100 uses 15% VRAM for inference, 78% for FP32+Adam training. — [[2025-07-24-gpu-calculator]]
