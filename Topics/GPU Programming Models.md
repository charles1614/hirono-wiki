# GPU Programming Models

## What

Abstractions above SIMT for authoring GPU code — CUDA, CUDA Tile, Triton, CUTLASS, cuTile Python.

## Current understanding

GPU programming models form a layered abstraction stack sitting above raw hardware ISA and SIMT execution. At the base, **CUDA C/C++** exposes the hardware directly: grids of thread-blocks, shared memory, warp-level primitives, and PTX intrinsics. Writing high-performance CUDA requires manual orchestration of tiling, register allocation, async memory copies (TMA/cp.async), and warp scheduling — tractable for library authors but hostile to ML researchers who need fast iteration. The fifth edition of *Programming Massively Parallel Processors* (2026-02-27) reflects this shift: new chapters on advanced matmul optimizations, wavefront parallelism, and multi-GPU NCCL/NVSHMEM signal that NVIDIA's own pedagogical leadership now treats warp specialization and TMA as required curriculum, not advanced-only appendices ([[2026-04-21-programming-massively-parallel-processor]]).

**Triton** occupies the middle of the stack. Its core abstraction is the *tile* — a statically-sized 2-D block of values that maps onto a warp group or CTA. The programmer writes scalar-looking loops over tiles; the MLIR-based compiler handles register layout, vectorization, shared-memory staging, and scheduling automatically. This makes attention kernels, matmul variants, and fused activation functions writable in ~100 lines of Python instead of 1,000 lines of CUDA, at 80–95% of hand-tuned throughput on common shapes. TensorRT-LLM's H200 path illustrates Triton's privileged status in the ecosystem: on Hopper, the OpenAI-shipped Triton MoE kernels are the *only* supported backend — CUTLASS MoE support on Hopper was "still ongoing" as of the 1.1.0rc1 release ([[2025-08-23-tensorrt-llm-docs-source-blogs-tech_blog]]). Triton's weakness is expressiveness at the extremes: non-rectangular tiles, fine-grained warp specialization, and producer–consumer pipelines between warp groups are difficult or impossible to express.

**CUTLASS** (and its successor **CuTe**) targets the other end of the spectrum: maximal performance for dense linear algebra where sub-1% headroom against roofline matters. CuTe introduces a *layout algebra* — composable rank-polymorphic descriptors for strides, tiling hierarchies, and register vs. shared memory placement — that makes it possible to write a single generic kernel spanning GEMM, batched GEMM, grouped GEMM, and attention variants. CUTLASS 3.x restructures the pipeline around warp-group MMA and TMA copy engines, separating producer (data fetch) and consumer (math) roles explicitly. Flux (ByteDance/PKU) chose CUTLASS over Triton specifically for Tensor-Core-heavy auto-tunable kernels — its tile-level kernel fusion maps each compute+communication tile pair to a single thread block, auto-tunable across GPU generations and interconnects ([[2025-10-09-flux-fast-software-based-communication-o]]). FlashMLA credits CUTLASS as one of its primary inspirations alongside FlashAttention's online softmax and Flash-Decoding's split-K ([[2026-01-28-flashmla-docs-20250422-new-kernel-deep-d]]). The trade-off: CuTe layouts are a DSL within C++ that demands understanding the mapping from logical coordinates to physical addresses at every level of the memory hierarchy.

**CUDA Tile** (introduced in CUDA 13.1, December 2025) is NVIDIA's explicit answer to the expressiveness gap between Triton and raw CuTe. NVIDIA frames it as "the largest and most comprehensive update to the CUDA platform since it was invented" — the claim is anchored by two components: **CUDA Tile IR** (a virtual ISA for tile-based GPU programming) and **cuTile Python** (the Python DSL surface). The core pitch is architectural: tile code is intended to be *portable to future GPU architectures* because the compiler abstracts hardware specialization (tensor cores, future units), whereas SIMT code requires the programmer to explicitly orchestrate thread-level partitioning ([[2026-01-08-nvidia-cuda-13-1-powers-next-gen-gpu-pro]]). As of the CUDA 13.1 release, cuTile is Python-only — C++ implementation is planned for a future release. Hardware scope covers Ampere / Ada / Blackwell (compute capability 8.x–12.x). Nsight Compute 2025.4 adds a dedicated Tile Statistics profiling section (Tile Mapping callouts, TMA utilization, Tile vs SIMT result-type column) with no equivalent for SIMT kernels ([[2026-01-08-nvidia-cuda-13-1-powers-next-gen-gpu-pro]]).

The practical division of labor as of 2026: Triton is the default for new ML kernels (fused attention, custom activations, MoE routing); CUTLASS/CuTe remains authoritative for GEMM and anything requiring fine-grained control over tile schedules and communication overlap; CUDA Tile / cuTile Python is an emerging portability-first path targeting algorithm-level tile operations where programmer-stated thread partitioning is the burden being removed. All compile through LLVM/NVCC to the same PTX/SASS layer — they differ in what invariants the programmer must state explicitly versus what the compiler or library infers. **SGLang's `jit_kernel` layer** ([[2026-03-16-sglang-claude-skills-add-jit-kernel-skil]]) occupies a niche between raw CUDA and Triton: it is CUDA C++ under the hood, but accessed through a project-specific abstraction kit (`TensorMatcher` for tensor validation, `AlignedVector` for 128/256-bit vectorized loads, `LaunchKernel` RAII for error-checked kernel dispatch, `tile::Memory` for cooperative 1D strided access). The system's JIT vs AOT split mirrors the Triton vs CUTLASS trade-off at the framework level — CUTLASS-dependent kernels graduate to the AOT `sgl-kernel` wheel-built path; everything lighter stays JIT for rapid iteration. PDL (SM90+) and persistent kernels (occupancy-capped grids) are explicitly taught, confirming that Hopper/Blackwell micro-architectural features are first-class concerns in the authoring layer.

A notable upstream pressure: AVO (NVIDIA, arXiv:2603.24517) demonstrated that LLM-coding agents running autonomous evolutionary search on B200 can beat cuDNN by 3.5% and FlashAttention-4 by 10.5% on MHA — implying that for the highest-performance kernels, the authoring layer is increasingly autonomous rather than hand-tuned ([[2026-03-31-avo框架实现gpu内核性能优化-小红书]]). The SKILL-pattern operationalization (SGLang's in-tree Claude Code / Codex SKILLs) extends this further into production-workflow territory: CUDA crash debug (PR #20910, staged API logging levels 1/3/5/10), YAML-driven benchmark search, and Torch Profiler analysis are all SKILL-wrapped, making high-experience operator workflows agent-executable. Practitioner evidence ([[2026-03-24-记录下sglang开发-优化-debug的技巧之大skill时代已来临]]) puts single-card speed gains of 40% (Z-Image) and 20%+ (Qwen-Image-2512) attributable to this SKILL-driven agent workflow.

## Comparison

| Axis | CUDA C/C++ | Triton | CUTLASS / CuTe | CUDA Tile / cuTile Python |
|---|---|---|---|---|
| **Abstraction level** | SIMT — thread-level grid/block/warp | Tile — statically-sized 2-D blocks over warp groups/CTAs | Layout algebra — rank-polymorphic stride + tiling descriptors | Tile IR (virtual ISA) + Python DSL — algorithm-level tile operations ([[2026-01-08-nvidia-cuda-13-1-powers-next-gen-gpu-pro]]) |
| **Primary language** | C/C++ | Python | C++ (templates) | Python (C++ planned for future release) ([[2026-01-08-nvidia-cuda-13-1-powers-next-gen-gpu-pro]]) |
| **Hardware scope** | All CUDA-capable GPUs | All CUDA-capable GPUs (auto-tuning configs per generation) | All CUDA-capable GPUs; 3.x optimized for Hopper/Blackwell TMA + warp-group MMA | Ampere / Ada / Blackwell (compute capability 8.x, 10.x, 11.x, 12.x) ([[2026-01-08-nvidia-cuda-13-1-powers-next-gen-gpu-pro]]) |
| **Architecture portability** | Low — programmer re-orchestrates thread layout per GPU generation | Medium — auto-tuning configs per generation; non-rectangular tiles and warp specialization require rewrite | Medium — template hierarchy abstracts precision + tile size; generation-specific paths are explicit | High (by design) — Tile IR abstracts hardware specialization; compiler handles tensor-core mapping ([[2026-01-08-nvidia-cuda-13-1-powers-next-gen-gpu-pro]]) |
| **Target workloads** | Any kernel; library authors; hardware bring-up | New ML kernels: fused attention, custom activations, MoE routing ([[2026-04-26-大模型推理八股-小红书]]) | Dense linear algebra, GEMM, comm+compute fusion (Flux), attention ([[2025-10-09-flux-fast-software-based-communication-o]]) | AI algorithms first; persistent kernels; pipelined attention prototyping ([[2026-01-08-nvidia-cuda-13-1-powers-next-gen-gpu-pro]]) |
| **Warp specialization / producer–consumer pipelines** | Full control — programmer-specified | Difficult / not expressible in current Triton | First-class — CUTLASS 3.x separates producer (TMA copy) and consumer (warp-group MMA) roles explicitly ([[2026-01-28-flashmla-docs-20250422-new-kernel-deep-d]]) | ? (too new for established guidance) |
| **Learning curve** | High — manual tiling, register allocation, async copies, PTX intrinsics | Low–medium — compiler handles register layout, vectorization, shared-memory staging | High — CuTe layout algebra is a DSL within C++; requires mapping logical ↔ physical addresses at every memory hierarchy level | Low (Python DSL); ? for Tile IR depth |
| **Profiling tooling** | Nsight Compute (full SIMT support) | Nsight Compute (source-mapping via Triton-MLIR) | Nsight Compute | Nsight Compute 2025.4 Tile Statistics: Tile Mapping callouts, TMA utilization, Tile vs SIMT result-type column ([[2026-01-08-nvidia-cuda-13-1-powers-next-gen-gpu-pro]]) |
| **Deployment precedents** | Universal | TensorRT-LLM H200 Triton MoE backend (only path on Hopper as of 1.1.0rc1) ([[2025-08-23-tensorrt-llm-docs-source-blogs-tech_blog]]) | TensorRT-LLM B200/GB200 CUTLASS MoE backend (pure EP only); Flux comm+compute fusion; FlashMLA ([[2025-08-23-tensorrt-llm-docs-source-blogs-tech_blog]], [[2025-10-09-flux-fast-software-based-communication-o]]) | Prototype / emerging — no production deployment evidence in Sources |
| **Multi-CTA cluster tiling** | Manual — programmer-specified | Requires retuning or rewrite across GPU generations | Template hierarchy explicitly handles multi-generation tiling | ? — open question; cuTile Python too new for established guidance ([[2026-01-08-nvidia-cuda-13-1-powers-next-gen-gpu-pro]]) |
