# Decoding Optimization

## What

Techniques for accelerating autoregressive decoding — speculative sampling, prefix caching, kernel-level scheduling.

## Current understanding

Autoregressive decoding is the central throughput bottleneck in LLM inference: generating one token at a time, with the KV cache growing linearly and each decode step memory-bandwidth-bound at small batch sizes. The techniques that address this bottleneck split into three largely orthogonal tracks — **speculative token generation**, **prefix cache reuse**, and **kernel-level scheduling** — each attacking a different layer of the pipeline. [[2026-05-08-a-survey-of-llm-inference-systems]] provides the unifying taxonomy: every technique maps to one or more of three cross-cutting primitives — **load prediction**, **adaptive mechanisms**, and **cost reduction**.

**Speculative sampling** is the most-studied acceleration technique in the corpus. The basic mechanism — a draft model proposes tokens; the target model verifies them in parallel; agreement yields multi-token steps at single-step cost — is now textbook. The frontier has moved to *scaling laws for draft models*. [[2025-10-09-eagle-3-scalingupinference-acceleration-]] establishes that EAGLE-3's draft-model speedup scales smoothly with training compute (a relation that prior methods like EAGLE and EAGLE-2 couldn't unlock because their feature-prediction loss `l_fea` capped data utilization). The production-significant result: **40% throughput improvement at batch size 64** when integrated into SGLang — directly refuting the prior assumption that speculation only helps latency-constrained single-user settings. The three architectural changes enabling this are: dropping `l_fea` in favor of direct token prediction, multi-layer feature fusion (low/mid/high-level target features concatenated into a 3k-vector), and a "training-time test" trick that closes the gap between draft training distribution and inference distribution. The production minimal config from Ant Group's H20 deployment is illustrative: `NEXTN --speculative-num-steps 1 --speculative-eagle-topk 1 --speculative-num-draft-tokens 2` — simple enough to coexist with the existing kernel zoo without complicating CUDA graph capture [[2026-05-06-蚂蚁开源-x-sglang-meetup技术回放解读系列之面向deepseek系]].

**Prefix caching** has a well-understood goal — reuse KV blocks across requests sharing a prompt prefix — but system complexity emerges at the scheduling layer. Baidu's **AttentionStore** [[2026-04-01-拒绝-openclaw-成为-吞金龙虾-百度百舸打造极致-kv-cache-调度]] demonstrates that realizing the win requires two levels of coordination: a cluster-level scheduler with a global block-location index (routing requests to cache-hot nodes by factoring KV-cache locality, instance load, and node health) and a node-level 3-tier store (HBM / DRAM / SSD) with pipelined reads. The headline result — **6.2× TTFT reduction** vs SGLang's default cache policy on DeepSeek R1 671B at 64K context — is attributable to the pipelining trick (huge-page + pinned-memory enabling simultaneous DRAM→HBM and SSD→DRAM transfers, collapsing 3 serial steps into 2), not to faster interconnects. This aligns with NVIDIA's analytical finding [[2025-10-09-beyond-the-buzz-a-pragmatic-take-on-infe]] that provisioned datacenter bandwidth is already sufficient for KV cache transfer; the bottleneck is scheduling and storage-tier orchestration, not the link itself.

**Kernel-level scheduling** for MLA decode attention is where the most technically novel work in the corpus lives. [[2026-01-28-flashmla-docs-20250422-new-kernel-deep-d]] establishes that MLA decoding on H800 is **compute-bound, not memory-bound** — because DeepSeek does not tensor-parallel the decode instances, keeping `h_q = 128`, which places the compute-to-memory ratio above H800's ~258 FLOPs/byte crossover. The consequence is that FlashAttention-3's standard ping-pong schedule cannot apply: the 64×512 output matrix consumes half the SM's register file (32,768 of 65,536 registers), leaving no room for a second output buffer. The solution is a novel **"seesaw" schedule** — splitting the output matrix vertically (O_L / O_R) across two warpgroups operating on alternating KV blocks — that achieves CUDA-Core/Tensor-Core overlap with only one output matrix in registers. The production result is **80% Tensor Core utilization** on the throttled H800 peak, with fine-grained TMA-GEMM pipelining (64×576 K-blocks split into nine 64×64 TMA copies) eliminating remaining memory-latency gaps.

The three tracks compose but do not substitute. Speculative decoding improves tokens-committed-per-target-step; prefix caching reduces TTFT by avoiding KV recomputation; kernel optimization raises the ceiling on decode throughput per GPU. The [[2026-05-08-a-survey-of-llm-inference-systems]] framing of "load prediction / adaptive mechanisms / cost reduction" as the three cross-cutting primitives maps cleanly: speculation is an adaptive mechanism, prefix caching is cost reduction, and kernel tuning is cost reduction at the hardware-utilization layer.

The main unresolved question in the corpus is the interaction between **prefill-decode disaggregation** and all three tracks. [[2025-10-09-beyond-the-buzz-a-pragmatic-take-on-infe]] shows disaggregation wins concentrate in prefill-heavy traffic on large models, and that dynamic Ctx:Gen GPU ratio matching is essential — a fixed ratio is Pareto-suboptimal across latency regimes. The Ant production deployment [[2026-05-06-蚂蚁开源-x-sglang-meetup技术回放解读系列之面向deepseek系]] uses disaggregation as the outer architecture and runs speculation + kernel optimizations *within* the decode pool — the composition is operational but the Pareto tradeoffs are still being characterized.
