# [2025-05-10] vLLM V1 整体流程｜从请求到算子执行

## TL;DR

A top-down walkthrough of the [[vLLM]] V1 engine request lifecycle (introduced as default from v0.8.x), focusing on the dual-process async architecture that decouples CPU-heavy preprocessing from GPU scheduling to reduce latency, with flow diagrams and source links for v0.8.5.

## Key claims

- [[vLLM]] V1 splits work into two OS processes: Process 0 handles tokenization/detokenization/parameter validation; Process 1 handles scheduling and model inference — these run in parallel (vs sequential in V0), reducing CPU idle time.
- `LLMEngine` (offline) and `AsyncLLM` (online) both build on `EngineCore`; `SyncMPClient` bridges them via ZMQ sockets (`input_socket` for requests, `output_socket` for results, with a dedicated polling thread on the output side).
- `Processor.process_inputs()` handles parameter validation, structured output grammar checking, and tokenization before forwarding to `EngineCore` via `SyncMPClient.add_request()`.
- The scheduler inside `EngineCore` (Process 1) runs a continuous step loop: schedule → forward (GPU) → postprocess; KV cache allocation uses paged blocks managed by a block manager.
- V1 advantages over V0: more readable code, easier feature integration, default-on optimizations, lower overhead through dual-process async overlap.
