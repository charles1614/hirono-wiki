# [2026-04-03] 用 AI + 浏览器自动化（CDP）实战演练

## TL;DR

An xhs hands-on demo by **晓汝AI** of using **Codex + CDM (a Chrome DevTools Protocol MCP)** to scrape a conference website (AD/PD 2026 Alzheimer/Parkinson conference) — 857 presentation records across 3 paginated pages, output as structured JSON. The author's load-bearing claim: **the agent finds the pagination mechanism via DOM + network-request analysis (`?p=` query param)**, not via Selenium-style UI-click simulation. Total operator input: a single goal-statement; the rest is the agent reading DOM, evaluating scripts, listing network requests, and batch-fetching. This is the "AI moves into the browser tab" pattern that complements the Wayve-style telematic agent integration and BBuf's per-machine SKILLs — different surface area, same methodological move (agent-as-engineer).

## Key claims

**Tool chain**: Codex agent + **CDM** = a Chrome DevTools Protocol (CDP)-based MCP server exposing tools like `New Page`, `List Pages`, `Take Snapshot`, `Wait For`, `Evaluate Script`, `List Network Requests`. The MCP-tool naming pattern (`Cdm MCP MCP New Page`) is the visible call shape in the agent logs. CDP is the underlying browser-control protocol; CDM is the application that wraps it as MCP tools.

**The workflow** (per the author's narration):

1. Goal: "scrape this conference's presentation list, paginated, full structured fields"
2. Agent enters the page via `New Page`, takes a DOM snapshot
3. Agent inspects DOM structure → identifies the card layout + per-card fields (title / author / session)
4. Agent finds pagination — **NOT by clicking "next" but by inspecting network requests + URL structure**. Discovers `?p=N` URL parameter pattern.
5. Agent batch-fetches across pages via parallelized requests
6. Agent emits structured JSON: `{pages_fetched: 3, records_fetched: 36, total: 857}` (the body says 857 records; the JSON result shows pages_fetched=3 / records_fetched=36 / total=857 — likely the demo fetched first 3 pages out of a 24-page total, with 857 being the all-page count)

**The Selenium-vs-CDP framing** (author's takeaway): traditional UI-click automation (Selenium) is brittle and slow; CDP-via-MCP lets the agent operate at the **network + DOM layer** directly. Implication: agents doing browser-automation should consume CDP-style network-aware tools, not Selenium-style click-and-wait tools. Different abstraction tier; meaningfully better economics for structured-data scraping at scale.

**The methodological pattern**: the operator gives ONE goal sentence; everything else is agent-driven reasoning + tool calls. The author's framing: "AI 操作浏览器就很顺利" (AI driving the browser worked smoothly) — emphasis on the minimal operator input. This is the agent-as-engineer thesis applied to web-scraping.
