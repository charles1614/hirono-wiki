# [2026-03-30] AI 代码看不懂？我做了个 prompt 回放工具 — vibe-replay

## TL;DR

An xhs post by **Tuo** (`Tuo 在做vibe-replay`) introducing **`vibe-replay`** — an open-source (MIT, `npx vibe-replay`) **session-replay tool for Claude Code + Cursor CLI** that captures local AI-coding sessions and renders them as a timeline of prompts → tool calls → diffs with per-turn analytics + cost / token statistics + AI-translated commentary. Built solo in one week, macOS-only at launch. **The author's framing problem**: "Cursor / Claude Code's built-in session tools are unusable; sharing vibe-coded sessions with teammates is a constant struggle." Output paths: GitHub Gist publish for online sharing, plus local export to SVG / Markdown / HTML / JSON for PR attachments or email. Demonstrates the **client-side observability layer** for agent-coded development — complements server-side trace observability ([[2026-04-01-openclaw-observability-基于-duckdb-构建-open]] DuckDB).

## Key claims

**Tool surface** (from `npx vibe-replay` dashboard):

- Captures sessions from **Claude Code + Cursor CLI** (also OpenCode integration). Headless-mode AI feedback for both.
- **Stats per dashboard** (demo state): SESSIONS 115, REPLAYS 20, TURNS 1453, TOOL CALLS 15,831. Claude Code is 94% of activity (108 / 115 sessions); Cursor 6% (7 / 115).
- **Distribution paths**:
  - GitHub Gist (one-click publish → public replay URL hosted at vibe-replay.com)
  - Local export: SVG / Markdown / HTML / JSON
  - "AI 翻译和润色 ai 对话" — local agent translates / polishes session content before sharing (the practical use case: "don't get caught swearing at the AI when sharing with colleagues")
- **Replay viewer** renders: per-turn prompts, per-turn tool-call lists with counts, code-diff inline, cost + duration metadata.
- **Insights tab** computes session-level analytics (likely tool-call distribution, error rate, turn-time histogram).

**Stated motivation**:
1. Native session-export from Claude Code / Cursor is "实在太难用了" (genuinely unusable) for sharing.
2. Comments + AI-translated polish let the author share their actual rough-edits with colleagues without leaking irritation.
3. Replay-as-artifact lives in the PR description (SVG / HTML) or in email — same surface as a code review.

**Why this matters for the wiki theme**: the agent-coding session is now a **first-class shareable artifact**, distinct from the code itself. vibe-replay treats the session as content; the **agent's reasoning trace becomes part of the PR**, not just its output. This parallels:

- [[2026-04-01-openclaw-observability-基于-duckdb-构建-open]] — server-side agent observability via DuckDB. Same problem (auditability of agent runtime), different side of the operator boundary.
- [[2026-04-29-welcome-to-learn-harness-engineering-lea]] — harness-engineering says "make agent state durable + reviewable." vibe-replay is the per-session-replay side of that doctrine.

**Operating constraints**:

- **Mac-only** at launch (xhs body says "暂时只支持 mac").
- **Open source, MIT, free**. GitHub: search "vibe-replay".
- One developer, one week of building. Modest commitment level vs production-grade tools, but the dashboard screenshots look polished.

Together they bound the agent-runtime-auditability problem from both sides: vibe-replay is per-developer-session capture for sharing + retrospective; OpenClaw-Observability is fleet-level audit + security scanning. Adds Tuo (`vibe-replay`) to the corpus's tracked-tooling list. If the project survives 6 months (real adoption signal for one-week solo builds), worth re-fetching with usage metrics + feature evolution.
