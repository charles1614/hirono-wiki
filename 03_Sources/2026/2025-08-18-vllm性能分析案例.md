# [2024-09-07] vLLM 性能分析案例

## TL;DR

A step-by-step profiling walkthrough showing how apparent AllReduce slowness in vLLM (0.5.1, TP=2, FP8, DeepSeek-6.7B, batch=1 on L20) was traced through several misleading symptoms to its root cause: CPU-side Python scheduling overhead blocking GPU kernel launch.

## Key claims

- Initial symptom: `cross_device_deduce_1stage` consumed ~50% of decode timeline on process 1 — appeared to be slow AllReduce, but was actually a false impression from process imbalance.
- Root cause: Process 2 was slower overall, and because AllReduce must complete on all ranks simultaneously, process 1 appeared stalled. AllReduce itself was fine.
- The actual delay in process 2 was not kernel-to-kernel scheduling overhead (another false conclusion) but CPU Python operations blocking kernel launch — visible in torch profile where GPU timeline had vertical CPU-GPU launch lines instead of the CPU running far ahead.
- Fix: CUDA Graph eliminated the CPU scheduling bottleneck; shown to remove the inter-kernel gaps.
- Second performance gap: step-to-step overhead from vLLM's serial scheduling/input/output pipeline (fixed in 0.5.4 with three-process async architecture).
- Third issue: FP8 dequantization kernels appearing un-fused in timeline — optimal fix is fusing them with preceding operators.
- Key lesson: "眼见不一定为实" — profile-based reasoning must trace to lowest-level cause, not stop at first plausible explanation.
