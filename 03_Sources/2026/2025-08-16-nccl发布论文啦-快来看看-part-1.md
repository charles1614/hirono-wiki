# [2025-07-20] NCCL发布论文啦？快来看看 (part-1)

## TL;DR

Deep walkthrough of the paper "Demystifying NCCL: An In-depth Analysis of GPU Communication Protocols and Algorithms" (NCCL v2.19.1). Covers NCCL API design, channel-based parallelism, three communication protocols (Simple/LL/LL128), and intra-/inter-node transport layer details including GPUDirect RDMA, QP layout, and flush semantics.

## Key claims

- NCCL organizes all communication through a communicator abstraction analogous to MPI, with `ncclCommInitAll` for single-process multi-GPU and `ncclCommInitRank` for cross-process communicators.
- Channels map to individual CUDA SMs (one block per channel), allowing parallel processing of data chunks across multiple channels; channel count is reduced heuristically for small messages to avoid underutilized 512KB FIFO buffers.
- Three protocols trade off latency and bandwidth: Simple uses memory fences for correctness at large message sizes; LL uses 4B data + 4B flag atomic units for low latency at small messages (25–50% peak BW); LL128 uses 120B data + 8B flag (128B atomics required) achieving ~95% peak BW on NVLink.
- NCCL's tuning model selects algorithm–protocol pairs at runtime based on topology, message size, and bandwidth; LL/LL128 for small messages, Simple for large.
- P2P\_DIRECT mode (same-process) bypasses IPC handles and intermediate FIFO buffers, writing directly between source and destination buffers, using head+tail counters for ordering.
- SHM transport is preferred over cross-socket PCIe P2P when inter-socket QPI/UPI traversal degrades P2P efficiency.
- IB Verbs transport uses two RC QPs per rank pair: a Forward QP for bulk RDMA\_WRITE + RDMA\_WRITE\_WITH\_IMM and a Reverse QP for CTS (clear-to-send) control signals, isolating latency-sensitive control flow from data bandwidth.
- Loop-back RDMA\_READ to a dedicated Flush QP serves as an efficient PCIe fence to drain outstanding RDMA writes into GPU memory, exploiting RDMA ordering: READ must complete after all prior WRITEs.
