# [2025-05-21] NCCL 系列之深入解析 NCCL 通信路径计算和优化

## TL;DR

AI闲谈公众号对 NCCL 2.26.3-1 通信路径计算机制的深度源码解析，涵盖 initTransportsRank 核心流程、拓扑 BFS 路径建模、PXN 代理路径、NVLink Sharp/NVLS 支持，以及 AllGather1/AllGather3 的协商流程。

## Key claims

- `initTransportsRank` 是 [[NCCL]] 初始化的核心函数，包含两个关键 AllGather：AllGather1 收集每 rank 的 ncclPeerInfo（rank、cudaDev、busId、hostHash、fabricInfo 等），AllGather3 交换详细算法图参数和拓扑排序，直接用于 Ring/Tree 通信拓扑构建。
- 路径类型层级（带宽从高到低）：PATH_LOC（本地）→ PATH_NVL（NVLink 一跳）→ PATH_NVB（NVLink 多跳）→ PATH_PIX（PCIe 同 Switch）→ PATH_PXB（跨 PCIe Switch）→ PATH_PHB（经过 CPU）→ PATH_NET（跨节点）。
- `ncclTopoComputePaths` 使用广度优先搜索（BFS）为所有 GPU/NIC/NVSwitch 节点预计算最优通信路径表：路径判优取带宽最大、跳数最少；不支持 P2P 时强制绕行本地 CPU。
- **PXN**（Proxy X NIC）：当 GPU 与目标 NIC 连接不优（跨多 PCIe 跳）时，NCCL 通过 NVLink 连接的代理 GPU 中转，优化轨道（rail）流量，避免进入 Spine Switch。
- NVLink Sharp（NVLS/NCCL_ALGO=NVSL）需启动 `nvidia-fabricmanager`，NCCL 2.17+ 在 Hopper（H100）NVSwitch 上支持；FM 负责配置 NVSwitch 端口路由、初始化 GPU、监控 NVLink/NVSwitch 错误。
- GDR（GPUDirect RDMA）不支持时，路径强制绕行本地 CPU（`ncclTopoCheckGdr` + `addInterStep`）；MNNVL 场景下 fabricInfo 携带 cluster UUID 和 cliqueId。
