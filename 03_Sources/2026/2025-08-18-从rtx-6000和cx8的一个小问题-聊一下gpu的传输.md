# [2025-08-07] 从 RTX 6000 和 CX8 的一个小问题，聊一下 GPU 的传输

## TL;DR

Starting from the RTX Pro + ConnectX-8 architecture question, this article explains [[NCCL]]'s three communication protocols (Simple, LL, LL128), intra-node GPU transfer mechanisms (P2P, SHM, NVLS), and inter-node transport (Socket, IB Verbs, GPUDirect RDMA).

## Key claims

- RTX 6000 + CX8 architecture: all intra-node and inter-node GPU communication routes through 400G NIC; ConnectX-8 acts as both NIC and PCIe switch, providing up to 50 GB/s per GPU IO bandwidth for intra-cluster communication.
- NCCL has three protocols optimized for different tradeoffs: Simple (near-peak bandwidth, ~6 µs hop latency, for large messages), LL (~1 µs latency, 25–50% bandwidth, flag-based sync, no GPU Direct RDMA), LL128 (~95% bandwidth, ~2 µs, 120B data + 8B flag, requires atomic 128B writes — auto-disabled on PCIe).
- Intra-node: NCCL prefers GPUDirect P2P over NVLink; falls back to PCIe P2P or SHM (host-memory relay) when P2P is unavailable or cross-socket P2P degrades. P2P_DIRECT avoids IPC handles and intermediate FIFO buffers when both ranks are in the same process.
- Inter-node: IB Verbs with GPUDirect RDMA (GPU directly DMA-accessible from NIC via nv_peer_mem / Linux DMA-BUF) eliminates host memory staging; requires GPU and NIC on same PCIe switch.
- Multi-channel IB: 2 logical channels per remote GPU×NIC pair (`NCHANNELS_PER_NET_PEER`), round-robin load balancing across QPs, improving utilization and ECMP path diversity.
- QP layout: forward QP for data (RDMA_WRITE + completion), reverse QP for CTS control messages — decoupled to avoid data/control contention.
- Local flush mechanism: after GPUDirect RDMA receive, a dummy self-directed RDMA_READ on a flush QP enforces PCIe write ordering without leaving the host.
