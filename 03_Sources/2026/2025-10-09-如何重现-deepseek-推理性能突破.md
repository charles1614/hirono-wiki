# [2025-05-19] 如何重现 DeepSeek 推理性能突破

## TL;DR

Alibaba's intelligent engine team describes how they reproduced [[DeepSeek-V3]]'s inference performance on [[RTP-LLM]] over two months, reaching Prefill 42.6K TPS/node and Decode 14.7K TPS/node on Alibaba Cloud RoCE (vs DeepSeek's 32.2K/14.8K on IB). Key techniques: PD disaggregation with mixed TP specs, full MicroBatch overlap with [[DeepEP]], MTP speculative decoding, and CUDA Graph planning.

## Key claims

- Test setup: Prefill 4-node/32-card EP=32, Decode 18-node/144-card EP=144 (128+16 redundant), TP=1, DP=EP=card count; 4K input/2K output at 1.6s TTFT / 50ms ITL constraints — 272 GPUs total.
- Combine phase carries FP16 tensors vs Dispatch's FP8, so Combine communication is ~2× longer; overlap strategy must allocate more compute time to hide Combine; recommendation: use quantized communication in inference to equalize phases.
- MTP (multi-token prediction) speculative sampling halves the required batch size to achieve the same GEMM arithmetic intensity; MTP offsets the latency increase from MicroBatch, making both optimizations synergistic.
- EPLB load balance is highly dataset-sensitive; test data doesn't represent production load; dynamic EP load balancing remains an open research direction.
- Framework overhead: ~1.5 ms inter-step Host overhead (Dynamic Batch), ~2 ms Kernel Launch overhead; CUDA Graph can eliminate launch overhead though adds architectural complexity; even when launch speed >> kernel execution speed, Device-side launch overhead persists.
- Intra-cluster dual-uplink fixes via NVSHMEM message-level (Normal kernel) and queue-level (Low Latency kernel) load balancing achieved communication performance matching or exceeding single-uplink IB; Low Latency mode latency reduced by 60%+ after traffic pattern optimization.
- 671B model weights loaded in minutes via: pre-processed weight format conversion (split/transpose offline), Direct IO + CUDA Pinned Memory bypassing page cache.
