# [2024-12-22] 聊聊 GPU 监控那些事：利用率 & 故障等

## TL;DR

Comprehensive GPU monitoring guide covering DCGM metric semantics, utilization indicators (MFU/HFU, GR_ENGINE_ACTIVE, SM_ACTIVE, SM_OCCUPANCY, Tensor Active, NVLink BW), and a taxonomy of GPU failure types (Xid errors, SXid/NVSwitch errors, memory row remap), illustrated with production war stories including fail-slow, GC straggler, fabricmanager OOM hang, and hardware Xid events.

## Key claims

- `DCGM_FI_PROF_GR_ENGINE_ACTIVE` supersedes `DCGM_FI_DEV_GPU_UTIL` — higher precision and MIG-compatible; a high value doesn't imply good efficiency (a 1-thread kernel yields 100% GPU Util, 2.5% SM Active on a 40-SM GPU).
- MFU = estimated FLOPs / hardware peak FLOPs (excludes recompute); HFU includes gradient checkpointing overhead; for A100/A800 clusters LLM MFU typically reaches 50–60%, H100/H800 under 50%.
- `DCGM_FI_PROF_PIPE_TENSOR_ACTIVE` approximates HFU upper bound for BF16 LLM training since most FLOPs flow through Tensor Cores; experimentally validated at ~48% Tensor Active vs 45.5% Megatron-LM MFU on a 2×8×H100 3B run.
- NVLink SHARP (NVLS) AllReduce achieves 480 GiB/s busbw vs 363 GiB/s without, but per-GPU NVLink BW drops (100–130 vs 170–190 GiB/s) by offloading reduce to NVSwitch broadcast/reduce.
- Xid 79 (GPU fallen off PCIe bus) co-occurs with PCIe errors in 43–63% of cases across two Meta clusters (RSC-1/RSC-2); NVLink errors account for 42.57% of Xids in DeepSeek Fire-Flyer cluster.
- Lemon node detection at Meta identified 1.2% of RSC-1 nodes (40/2000 A100 nodes) responsible for 13% of daily job failures; evicting them cut large-job (512+ GPU) failure rate from 14% to 4%.
- GPU memory row remap (`DCGM_FI_DEV_ROW_REMAP_PENDING`) can cause silent loss divergence before surfacing as job crash; requires DCGM level-3 diagnostic (exclusive access) to detect.
- Python GC holding the GIL caused periodic straggler events in multi-rank training; Megatron-LM now offers explicit coordinated GC at fixed step intervals.
- nvidia-fabricmanager kill (via OOM) on H100 causes NCCL AllReduce hangs because NCCL defaults to NVLS algorithm; workaround: `NCCL_ALGO=Ring` or restart fabricmanager.

## What this changes

- Tensor Active as practical HFU proxy for clusters lacking MFU instrumentation.
- Coordinated GC and fabricmanager watchdog as must-have for large H100 deployments.
