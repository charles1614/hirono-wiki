# [2025-11-08] PyTorch Symmetric Memory：解锁NVLink可编程性的新型分布式计算范式

## TL;DR

Technical deep-dive on [[PyTorch Symmetric Memory]], introduced as an experimental feature in PyTorch 2.9, which abstracts NVLink's virtual address mapping so developers can write custom multi-GPU kernels with direct peer memory access, enabling intra-kernel communication, one-shot all-reduce at ~500 GB/s effective bandwidth on 8×H100, and 1.3–1.5× speedup for fused all-gather + matmul kernels.

## Key claims

- Core API: `torch.ops.symm_mem.empty()` allocates a symmetric tensor; `rendezvous(tensor, group)` exchanges CUDA `cuMemMap`/`cuMemSetAccess` handles across the process group, binding remote buffers to local virtual address space; `barrier(handle)` uses atomicCAS on signal pads, compatible with CUDA Graph.
- Three programming opportunities: intra-kernel put/get operations overlapping compute and communication; sub-microsecond one-sided remote access without remote kernel cooperation; custom communication patterns (e.g., MoE all-to-all-v with non-standard data layouts).
- One-shot all-reduce leverages NVLink Switch (NVLS) multicast: all devices simultaneously read remote buffers and reduce in-place, reaching ~500 GB/s on 8×H100 for 1 GB tensors vs standard NCCL.
- Two-shot all-reduce: local reduce then broadcast, used when NVLS is unavailable.
- Copy Engine handles data movement in parallel with SM computation, reducing SM contention in all-gather kernels.
- Triton fused all-gather + matmul kernel demonstrates 1.3–1.5× speedup on Llama-70B scale workloads vs unfused approach.
- NVSHMEM backend extends symmetric memory to multi-node RDMA (InfiniBand/IB-GDA); CUDA backend targets single-node NVLink setups.
