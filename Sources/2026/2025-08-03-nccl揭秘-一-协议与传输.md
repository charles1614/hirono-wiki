# [2025-08-04] NCCL揭秘(一)——协议与传输

## TL;DR

A deep-dive into [[NCCL]] 2.19.1 internals based on the paper "Demystifying NCCL: An In-depth Analysis of GPU Communication Protocols and Algorithms." Covers the four API categories, three execution models, channel architecture, the three communication protocols (Simple/LL/LL128), and the full intra-node and inter-node transport stack.

## Key claims

- [[NCCL]] exposes four API categories: communicator management (`ncclCommInitAll`, `ncclCommInitRank`, `ncclCommDestroy`, `ncclCommAbort`), collective communication (AllReduce, Broadcast, Reduce, AllGather, ReduceScatter), point-to-point (`ncclSend`/`ncclRecv`), and group calls (`ncclGroupStart`/`ncclGroupEnd`).
- Channels are independent CUDA blocks on distinct SMs; input buffers are partitioned across channels for parallel throughput. Too many channels causes under-filled 512 KiB NIC FIFOs, degrading PCIe and network throughput — NCCL adaptively reduces `nChannels` for smaller messages via `calcP2pChannelCount`.
- **Simple protocol** maximizes bandwidth for large messages via memory fences (`IBV_SEND_FENCE` + doorbell write), at the cost of at least one extra RTT per notification.
- **LL protocol** eliminates fences: transmits 4B data + 4B flag = 8B via a single atomic store (`st.volatile.global.v4.u32`); receiver polls until both flag copies match. Uses host DRAM buffers (not GPU memory) to avoid slow PCIe polls; achieves only 25–50% peak bandwidth, suitable for latency-sensitive small messages.
- **LL128 protocol** extends LL to 128-byte granularity (120B data + 8B flag) exploiting NVLink's atomic 128B visibility guarantee; achieves ~95% peak bandwidth; requires hardware support and is disabled if atomicity cannot be guaranteed (e.g. some PCIe systems). NCCL defaults to LL128 on machines with NVLink.
- **Intra-node transport**: prefers NVLink GPUDirect P2P → PCIe GPUDirect P2P → Shared Memory (SHM). P2P_DIRECT mode (same process) uses direct GPU pointers, bypassing IPC handles and intermediate FIFOs; synchronized via atomic head/tail counters in `ncclSendMem`/`ncclRecvMem`.
- **Inter-node IB transport**: defaults to 2 logical channels per remote-GPU-NIC pair (controlled by `NCHANNELS_PER_NET_PEER`); 2 RC QPs per rank pair split data (forward QP, RDMA_WRITE + RDMA_WRITE_WITH_IMM) and CTS (reverse QP) to prevent HOL blocking.
- GPUDirect RDMA (GDR): GPU memory registered directly with NIC via `nv_peer_mem` or Linux DMA-BUF, eliminating host-memory staging; requires GPU and NIC to share the same PCIe switch.
- Loopback RDMA_READ flush: a dedicated "flush" QP connected to itself issues a zero-byte RDMA_READ after all writes, providing a cheap PCIe ordering barrier to ensure GPU kernels see completed writes.
