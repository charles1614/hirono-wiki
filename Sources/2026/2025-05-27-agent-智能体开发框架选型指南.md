# [2024-11-05] Agent 智能体开发框架选型指南

## TL;DR

Practical comparison of three AI agent implementation approaches — no-framework code, [[LangGraph]], and [[LlamaIndex]] Workflows — building the same multi-skill RAG/SQL/analytics agent in each, with concrete trade-offs for team and project factors.

## Key claims

- No-framework approach: the skill router (powered by OpenAI function calling) tends to self-generate SQL rather than delegating to the SQL skill — requires significant prompt engineering; output format handling across skills adds complexity, but the code is cleanest and easiest to understand for learning agent internals.
- [[LangGraph]] (released Jan 2024): built on LangChain, uses a Pregel graph structure with nodes, edges, and conditional edges to support cyclic flows; most structured and team-friendly but carries the heaviest framework constraints and debugging is hard if you disagree with its design philosophy.
- [[LlamaIndex]] Workflows: event-driven architecture that balances framework structure with developer freedom; inherent async design can add complexity in some scenarios; depends less on the framework than LangGraph.
- Framework selection hinges on three factors: existing framework integration depth in the project, team familiarity with agent architecture patterns, and availability of reference projects using similar approaches.
- The test agent required three skills — RAG over product docs, SQL generation over LLM-app data, and higher-level data analytics — with a Gradio chatbot UI.

## What this changes

Framework choice is a project-context decision, not a ranking: no-framework wins for understanding/simplicity, LangGraph for structured teams, LlamaIndex Workflows for flexibility without full custom code.
